1. Shallow Copy aur Reference ki dikkat (The Real Problem)
JavaScript mein Objects "Reference" se pass hote hain.

Maan lo payment ek dabba hai jo Broker se aaya hai. Agar tum direct payment ko Mongoose ke function mein bhej dete ho, toh tum wahi original dabba bhej rahe ho. Agar Mongoose us dabbe mein kuch badlav karta hai (jaise koi internal ID chipkana), toh tumhara original data bhi badal jayega.

Lekin jab tum { ...payment } likhte ho, toh tum ek naya dabba banate ho aur purane dabbe ka saara saaman nikaal kar is naye dabbe mein daal dete ho. Isse original data safe rehta hai.

2. Mongoose aur "Unwanted Data"
Jab Broker se payment data aata hai, toh kabhi-kabhi usme kuch extra cheezein hoti hain jo Database Schema ka hissa nahi hoti (jaise message metadata, timestamp, ya internal functions).

Agar tum direct payment likhte ho: Tum Mongoose ko poora ka poora "complex" object pakda rahe ho. Kabhi-kabhi Mongoose ko us complex object ko process karne mein confusion ho jati hai kyunki usme extra hidden properties ho sakti hain.

Agar tum { ...payment } likhte ho: Yeh process sirf "Data" properties ko uthata hai. Yeh hidden methods ya complex prototypes ko chhod deta hai. Mongoose ko ek Clean aur Plain Object milta hai, jo update karne ke liye perfect hai.

3. Ek Real-Life Example
Socho tumhare paas ek Form (payment) hai jo thoda ganda hai, uspe kuch purane stickers (internal metadata) lage hain.

payment (Direct): Tumne wahi ganda form seedha file mein laga diya.

{ ...payment } (Spread): Tumne ek naya saaf panna liya, aur puraane form ki saari kaam ki jaankari (Status, ID, Amount) naye panne par likh li aur naya panna file mein lagaya.

Naya panna (Spread operator) hamesha zyada safe aur professional hota hai programming mein.
